INTER-PROCESS COMMUNICATION

Learn how processes communicate and share data in modern operating systems — from shared memory to sockets and message queues.

What is IPC?

Inter-Process Communication (IPC) refers to the mechanisms an operating system provides to allow processes to communicate with each other and synchronize their actions. Since processes typically run in isolated memory spaces, IPC methods provide controlled ways for processes to exchange data and signals.

Why IPC is Essential

Modern computing relies heavily on IPC for everything from client-server applications to microservices architectures. IPC enables the modular design of software systems, where different components can work together while maintaining isolation.

Core Purposes of IPC

  • Data Transfer: Moving information between processes
  • Resource Sharing: Allowing multiple processes to access the same resources
  • Notification: Alerting processes about events or state changes
  • Process Control: One process influencing the execution of another
  • Synchronization: Coordinating activities between multiple processes

  • Shared Memory Model: Processes access a common region of memory to exchange data. High performance but requires careful synchronization.
  • Message Passing Model: Processes exchange discrete messages through a kernel-managed channel. More structured but may have higher overhead.
Note: Some IPC mechanisms combine elements of both models or offer variations tailored to specific use cases.

Common IPC Mechanisms

Shared Memory

Shared memory allows multiple processes to access the same block of memory. It's typically the fastest IPC method as data doesn't need to be copied between processes.

  • Advantages: Very high speed, especially for large amounts of data
  • Challenges: Requires synchronization mechanisms to prevent race conditions
  • Use cases: High-performance computing, multimedia processing

Pipes

Pipes provide a one-way flow of data between processes. Named pipes (FIFOs) allow unrelated processes to communicate.

  • Advantages: Simple to use, built into most shell environments
  • Limitations: Typically unidirectional, often limited to related processes
  • Use cases: Command-line pipelines, producer-consumer patterns

Message Queues

Message queues allow processes to exchange discrete messages in a structured format, providing asynchronous communication.

  • Advantages: Persistence of messages, prioritization, many-to-many communication
  • Limitations: More complex than pipes, potential overhead
  • Use cases: Task distribution systems, loosely coupled services

Sockets

Sockets provide a communication endpoint for sending and receiving data across a network or within a single system.

  • Advantages: Work across different machines, standard API, bidirectional
  • Limitations: More overhead than other local IPC methods
  • Use cases: Network applications, client-server architectures, distributed systems

Signals

Software interrupts sent to a process to notify it of important events. Used for simple notifications rather than data exchange.

  • Advantages: Lightweight, asynchronous, built into the OS
  • Use cases: Terminating processes, alerting about exceptions

Semaphores

Synchronization primitives used to control access to shared resources. They coordinate access but don't directly transfer data.

  • Advantages: Prevent race conditions, relatively simple
  • Use cases: Synchronizing shared memory access, mutual exclusion

Memory-Mapped Files

Files mapped directly into a process's address space, allowing processes to share data through the filesystem.

  • Advantages: Persistence, handles very large data sets efficiently
  • Use cases: Databases, caches, large data processing

IPC Implementation Examples

Shared Memory Example

Shared Memory — shmget / shmatC
#include <stdio.h>
#include <stdlib.h>
#include <string.h>
#include <sys/ipc.h>
#include <sys/shm.h>
#include <unistd.h>

#define SHM_SIZE 1024

int main() {
    key_t key = ftok(".", 'a');
    int shmid = shmget(key, SHM_SIZE, IPC_CREAT | 0666);
    char *shm_ptr = shmat(shmid, NULL, 0);
    
    pid_t pid = fork();
    
    if (pid == 0) {
        sleep(2);  // Wait for parent to write
        printf("Child reading: %s\n", shm_ptr);
        shmdt(shm_ptr);
    } else {
        strcpy(shm_ptr, "Hello from the parent process!");
        wait(NULL);
        shmdt(shm_ptr);
        shmctl(shmid, IPC_RMID, NULL);
    }
    return 0;
}

Pipe Example

Pipe — parent to child data transferC
#include <stdio.h>
#include <unistd.h>
#include <string.h>
#include <stdlib.h>

int main() {
    int pipefd[2];
    char buffer[100];
    const char *message = "Hello from parent via pipe!";
    
    pipe(pipefd);
    pid_t pid = fork();
    
    if (pid == 0) {
        close(pipefd[1]);  // Close write end
        int n = read(pipefd[0], buffer, sizeof(buffer));
        buffer[n] = '\0';
        printf("Child received: %s\n", buffer);
        close(pipefd[0]);
    } else {
        close(pipefd[0]);  // Close read end
        write(pipefd[1], message, strlen(message));
        close(pipefd[1]);
        wait(NULL);
    }
    return 0;
}

Message Queue Example

Message Queue — msgsnd / msgrcvC
#include <stdio.h>
#include <sys/ipc.h>
#include <sys/msg.h>
#include <string.h>
#include <stdlib.h>

struct message { long msg_type; char msg_text[100]; };

int main() {
    key_t key = ftok(".", 'b');
    int msgid = msgget(key, IPC_CREAT | 0666);
    struct message msg;
    pid_t pid = fork();
    
    if (pid == 0) {
        msgrcv(msgid, &msg, sizeof(msg.msg_text), 1, 0);
        printf("Child received: %s\n", msg.msg_text);
    } else {
        msg.msg_type = 1;
        strcpy(msg.msg_text, "Hello from parent!");
        msgsnd(msgid, &msg, strlen(msg.msg_text)+1, 0);
        wait(NULL);
        msgctl(msgid, IPC_RMID, NULL);
    }
    return 0;
}

Best Practices & Considerations

Choosing the Right IPC Mechanism

  • Relationship between processes: Parent-child relationships favor pipes; unrelated processes use named pipes, message queues, or sockets
  • Amount of data: For large transfers, shared memory is most efficient
  • Communication pattern: One-way, request-response, or complex messaging needs
  • Persistence requirements: Whether data needs to survive process termination
  • Location of processes: Same machine or different machines
Common Challenges with IPC:
  • Race Conditions: Multiple processes accessing shared resources without synchronization
  • Deadlocks: Processes waiting for each other indefinitely
  • Resource Management: Ensuring proper cleanup of IPC resources
  • Error Handling: Dealing with various failure scenarios
Beyond Local IPC: Technologies like RPC, message brokers (RabbitMQ, Kafka), and REST APIs extend IPC concepts to distributed systems across network boundaries.
