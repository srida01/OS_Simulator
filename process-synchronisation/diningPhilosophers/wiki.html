DINING
PHILOSOPHERS

Module 12 · Synchronization Problem · Wiki
01 / Overview

OVERVIEW

The Dining Philosophers Problem is a classic computer science thought experiment formulated by Edsger Dijkstra in 1965. It illustrates the fundamental challenges of resource allocation and process synchronization in concurrent systems.

The problem involves five philosophers seated around a circular table. Each philosopher alternates between thinking and eating, and each requires two chopsticks to eat — one from their left, one from their right.

Key Insight

The scenario models real OS scenarios where multiple processes compete for a limited set of shared resources, and a naive allocation strategy can lead to total system deadlock.

02 / Problem Description

PROBLEM DESCRIPTION

The setup is deceptively simple, yet captures the essence of multi-process resource contention:

  • Five philosophers sit around a circular table with a shared bowl of rice in the center.
  • Five chopsticks are placed between them — one between each adjacent pair.
  • A philosopher can only eat when they hold both the left and right chopstick.
  • They may only pick up one chopstick at a time.
  • After eating, they release both chopsticks and return to thinking.

The challenge: chopsticks are shared resources. Each chopstick sits between two philosophers and can only be held by one at a time.

03 / Goal

GOAL

Design a concurrent algorithm that:

  • Allows philosophers to eat whenever they can acquire both adjacent chopsticks.
  • Avoids deadlock — a state where all philosophers hold one chopstick and wait forever for the other.
  • Prevents starvation — ensuring no philosopher waits indefinitely to eat.
  • Maximizes concurrency — allowing multiple philosophers to eat simultaneously when possible.

Shared Data Structure

Each chopstick is represented as a semaphore, initialized to 1 (available):

semaphore chopstick[5] = {1, 1, 1, 1, 1}
04 / Pseudocode

PHILOSOPHER PROCESS

Each philosopher i runs the following loop indefinitely:

philosopher_process.c — pseudocode 
do {
    // Acquire left chopstick
    wait(chopstick[i]);

    // Acquire right chopstick
    wait(chopstick[(i + 1) % 5]);

    // ── Critical Section ──
    // eat for a while...

    // Release left chopstick
    signal(chopstick[i]);

    // Release right chopstick
    signal(chopstick[(i + 1) % 5]);

    // think for a while...

} while (TRUE);

The wait() call decrements the semaphore (acquires the resource). The signal() call increments it (releases the resource). If wait() is called on a semaphore with value 0, the process blocks until the resource is freed.

05 / Deadlock Issue

DEADLOCK ISSUE

The naive solution above has a critical flaw. Consider this scenario:

⚠ Deadlock Scenario

All five philosophers become hungry simultaneously and each picks up their left chopstick at the same time. Every semaphore drops to 0. Now each philosopher waits for their right chopstick — which is held by the neighbor to their right. No philosopher can proceed. The system is in total deadlock.

The four necessary conditions for deadlock (Coffman conditions) are all met:

  • Mutual Exclusion: A chopstick can only be held by one philosopher at a time.
  • Hold and Wait: Each philosopher holds one chopstick while waiting for another.
  • No Preemption: A chopstick cannot be forcibly taken from a philosopher.
  • Circular Wait: Each philosopher waits for the resource held by the next one, forming a cycle.
06 / Handling Strategies

DEADLOCK HANDLING STRATEGIES

Several strategies exist to break one or more of the Coffman conditions and prevent deadlock:

Strategy 01

LIMIT CONCURRENCY

Allow at most four philosophers to sit at the table simultaneously. This breaks the circular wait condition — at least one philosopher can always obtain both chopsticks.

Strategy 02

ATOMIC PICKUP

Allow a philosopher to pick up both chopsticks only if both are available, inside a critical section. This prevents partial resource acquisition and eliminates hold-and-wait.

Strategy 03

ASYMMETRIC ORDER

Odd-numbered philosophers pick up the left chopstick first; even-numbered ones pick up the right first. This asymmetry breaks the circular wait condition.

✓ Best Practice

In real operating systems, resource ordering (Strategy 3) is the most commonly applied technique — processes must always request resources in a globally defined order, preventing circular dependencies.

Strategy 1: Limiting Concurrency

Allow at most N-1 philosophers (4 out of 5) to attempt eating simultaneously. A semaphore initialized to 4 guards the entry:

limit_concurrency.c — pseudocode 
semaphore table = 4;
semaphore chopstick[5] = {1, 1, 1, 1, 1};

do {
    wait(table);           // Limit to 4 simultaneous diners
    wait(chopstick[i]);    // Get left chopstick
    wait(chopstick[(i+1)%5]); // Get right chopstick

    // ... eat ...

    signal(chopstick[(i+1)%5]);
    signal(chopstick[i]);
    signal(table);        // Release table seat
} while (TRUE);

Advantage: Simple to implement. Disadvantage: Underutilizes resources — we never use all 5 chopsticks simultaneously, even though it's theoretically possible.

Strategy 2: Atomic Acquisition

Use a monitor or mutex to ensure both chopsticks are acquired atomically. If a philosopher cannot acquire both, they release the first and wait:

atomic_acquisition.c — pseudocode 
semaphore mutex = 1;
semaphore chopstick[5] = {1, 1, 1, 1, 1};

do {
    wait(mutex);
    // Critical section: check both resources
    if (chopstick[i] == 1 && chopstick[(i+1)%5] == 1) {
        chopstick[i] = 0;
        chopstick[(i+1)%5] = 0;
        signal(mutex);
        // ... eat ...
        chopstick[i] = 1;
        chopstick[(i+1)%5] = 1;
    } else {
        signal(mutex);
        // Retry after brief delay
    }
} while (TRUE);

Advantage: No hold-and-wait. Disadvantage: Busy-waiting or spinning wastes CPU cycles; philosophers may suffer starvation if unlucky with timing.

Strategy 3: Resource Ordering (Asymmetric Order)

Break the circular wait by enforcing a global resource order. Philosophers with even IDs pick up their right chopstick first; odd-numbered ones pick up their left first:

asymmetric_order.c — pseudocode 
semaphore chopstick[5] = {1, 1, 1, 1, 1};

// Philosopher i
do {
    if (i % 2 == 0) {
        // Even philosopher: pick RIGHT first, then LEFT
        wait(chopstick[(i+1)%5]);
        wait(chopstick[i]);
    } else {
        // Odd philosopher: pick LEFT first, then RIGHT
        wait(chopstick[i]);
        wait(chopstick[(i+1)%5]);
    }

    // ... eat ...

    signal(chopstick[i]);
    signal(chopstick[(i+1)%5]);

} while (TRUE);

Advantage: Simple, no busy-waiting, no artificial concurrency limits. Implementation: This is the most widely used approach in practice.

Why It Works

With asymmetric ordering, there is no circular dependency. Philosopher 0 (even) waits for chopsticks in order [right, left], while Philosopher 1 (odd) waits for [left, right]. The global order imposed prevents cycles, ensuring at least one philosopher can always make progress.

Strategy 4: Monitor-Based Solution

The most elegant solution uses a monitor — a high-level synchronization construct combining a mutex and condition variables. Each philosopher's state (thinking, hungry, eating) is tracked, and they eat only when both neighbors are not eating:

monitor_solution.c — pseudocode 
monitor DiningPhilosophers {
    enum State {THINKING, HUNGRY, EATING};
    State state[5] = {THINKING, THINKING, THINKING, THINKING, THINKING};
    condition self[5];

    void pickup(int i) {
        state[i] = HUNGRY;
        while (state[(i+4)%5] == EATING || state[(i+1)%5] == EATING) {
            self[i].wait();  // Wait until neighbors finish
        }
        state[i] = EATING;
    }

    void putdown(int i) {
        state[i] = THINKING;
        self[(i+4)%5].signal();  // Wake left neighbor
        self[(i+1)%5].signal();  // Wake right neighbor
    }
}

// Philosopher process
do {
    DiningPhilosophers.pickup(i);
    // ... eat ...
    DiningPhilosophers.putdown(i);
    // ... think ...
} while (TRUE);

Advantage: Guarantees deadlock-freedom and near-starvation-freedom when combined with fairness policies (e.g., FIFO waiting queues). Cleaner abstraction: No need to reason about semaphore ordering.

✓ Recommended Approach

The monitor-based solution is the gold standard in modern OS design. Languages like Java use monitors (synchronized blocks + wait/notify) and modern systems employ higher-level primitives like condition variables in pthread mutexes.

Comparison Summary

Each strategy trades off simplicity, efficiency, and fairness:

  • Concurrency Limiting: Simple but inefficient.
  • Atomic Acquisition: Fair but prone to busy-waiting and starvation.
  • Resource Ordering: Efficient and deadlock-free; most practical for low-level implementation.
  • Monitor-Based: Elegant and fair; preferred in higher-level languages and modern OS kernels.
